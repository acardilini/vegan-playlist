# Session 3.3 — Remaining Pages & Polish: Design

_Approved by the curator 2026-07-11 (brainstorming session). Executes the last session
of Phase 3 (Brand & UI Rebuild); see `docs/PROJECT_PLAN.md`._

## Goal

Bring the remaining pages onto the Session 3.1 design system, fix the site-wide 390px
horizontal overflow, and do a pragmatic accessibility pass — closing Phase 3 so Phase 4
(deployment) can start from a fully branded, responsive site.

## Scope

**In scope (frontend-only):**
- Restyle: Playlists, Playlist Detail, Submit (`SongSubmissionForm`), Dashboard
  (`DataDashboard`), About.
- Public playlists become **read-only** (curator decision, this session — see below).
- Fix the pre-existing site-wide horizontal overflow at 390px **before** the page work.
- Pragmatic accessibility pass across public pages.
- Light-touch admin cleanup: fix visible clashes/broken spots only.
- App.css bridge cleanup: delete each restyled page's legacy block (+ emptied bridge
  `:root` entries) as it lands.

**Out of scope:**
- Full admin restyle (deferred — admin is local-only, already on-palette via the 3.1
  bridge, and changes again in Phase 4 with real auth).
- New features, backend/API/DB changes. The playlist API routes are untouched: the
  admin Manage Playlists tab consumes the same public endpoints and must keep working.
- Formal WCAG audit (pragmatic pass only).

## Curator decisions (this session)

1. **Admin: light touch.** No tab-by-tab restyle; fix what visibly clashes or is broken.
2. **Public playlists: read-only.** Remove the public "Create New Playlist" button +
   modal and the add-to-playlist UI from public song surfaces; remove remove-song
   controls from the public playlist detail view. Playlists are curator-made collections
   until real auth lands (Phase 4+, per the 2026-07-07 deferral decision). Backend
   endpoints stay (admin tab uses them).
3. **Sequencing: shell-first** (approach A) — fix the 390px overflow before restyling
   pages, so each page is verified at mobile width as it lands.

## Plan of work

### 1. Shell fix (first)

Diagnose the root cause of the horizontal overflow at 390px — it affects every route
including the untouched About page, so it is shell- or page-container-level (suspects:
fixed widths / `100vw` / min-widths in the header, footer, or container rules still in
App.css). Fix at the root; no `overflow-x: hidden` band-aid unless the root cause
genuinely demands it. Verify every route at 390px before page work starts.

### 2. Page restyles

Per page: restyle → delete its legacy App.css block (+ bridge entries emptied by it) →
verify at 1280 and 390.

- **Playlists (`pages/PlaylistsPage.jsx`)** — browse-only grid of kit-style cards
  (name, song count, description, creator, date). Striped placeholder artwork replaces
  the dead Spotify-green `via.placeholder.com` image. `CreatePlaylistModal` and the
  create flow are deleted from the page.
- **Playlist Detail (`pages/PlaylistDetailPage.jsx`)** — read-only song list styled
  like the artist-page album rows, fixing the inherited `.song-item` 40px-thumb clash
  noted in 3.2. Remove-song controls deleted from the public view.
- **Add-to-playlist removal** — `AddToPlaylistModal` usage removed from public song
  surfaces (song cards / song detail); component deleted if nothing else uses it.
- **Submit (`components/SongSubmissionForm.jsx`)** — form restyled to the kit mockup
  using existing `.input/.select/.btn` classes; same fields, same endpoint.
- **Dashboard (`components/DataDashboard.jsx`)** — restyled to the kit mockup, stats
  fed live from the existing analytics endpoints. If any charts are (re)styled, follow
  the dataviz skill with brand tokens.
- **About (`pages/AboutPage.jsx`)** — adopt the kit's copyedited content and layout
  (sentence-case headers, "Submit a song" secondary button linking to `/submit`);
  stat badges fed by live stats rounded down to the hundred, matching the home hero
  (the mockup's hardcoded "1,208+ / 556+" are stale).

### 3. Accessibility pass (public pages)

- Visible focus states on all interactive elements (tokens define a focus ring).
- Form inputs labelled; images have meaningful alt text; heading levels in order.
- Keyboard walk of the main flows: search → song page; playlists → playlist detail;
  submit form end-to-end.

### 4. Admin light touch

Fix visible clashes/broken spots encountered while checking tabs (e.g. rules leaking
between public and admin styles). No structural changes; admin App.css blocks stay.

## Error handling / data flow

No API or data-flow changes. Existing loading/error/empty states are restyled to the
shared `components.css` message classes (already used by the 3.2 pages).

- `npm run build` clean; eslint 0 errors on touched files.
- End-Session guide: update `PROJECT_STATE.md` / `PROJECT_PLAN.md` (Phase 3 exit
  criteria met), commit, push.
